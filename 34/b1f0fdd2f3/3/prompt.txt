## Remaining Issues (Minor)

### 1. Tag display localization: call-site responsibility
The spec lists `TagList`, `TagInput`, `GlassLikeTagInput` as needing localized display (Section 7, File Changes table). These are **generic atom components** that receive `Array<string>` — they don't know about `Tag` objects or languages. Localization should happen at call sites:
- `MetadataFormFields.tsx` converts `RecipeMetadataWrite.tags` → localized display strings
- `useTagSuggestions.ts` returns localized labels in `TagSuggestion.label`
- Components rendering recipe cards resolve `tagIds` → localized labels before passing to `TagList`

The spec's `getTagDisplayLabel(tag, uiLanguage)` helper is correct in concept; it just needs to be called at the right layer (not inside atoms).

### 2. `useGlobalOptions` route type
The spec doesn't show the exact type change. Currently line 12:
```typescript
route: "/manage-books" | "/about";
```
Needs to become:
```typescript
route: "/manage-books" | "/about" | "/settings";
```

### 3. Polish pluralization edge cases
Polish has 3 plural forms (`_one`, `_few`, `_many`). The spec configures `compatibilityJSON: "v4"` which is correct. Edge cases to verify during implementation:
- `count=0` → should use `_many` ("0 przepisów")
- `count=1` → `_one` ("1 przepis")
- `count=2-4` → `_few` ("2 przepisy")
- `count=5-21` → `_many` ("5 przepisów")
- `count=22-24` → `_few` ("22 przepisy")

## Overengineering Analysis

### Recommend cutting: `Recipe.manuallyEdited` tracking
- **What**: New boolean field on `Recipe`, change detection in `updateRecipe`, filtering in reprocessing
- **Why cut**: Only used when changing a book's language (rare action). For V1, reprocess ALL recipes in the book — users can re-edit if needed.
- **Impact**: Removes ~30% of Phase 6 complexity, eliminates a new Recipe field, and simplifies RecipesContext

### Recommend simplifying: Manage-books language UI
- The spec proposes both a swipe globe button AND an inline badge alternative
- **Recommendation**: Use only the inline badge ("EN" / "PL" in subtitle, tappable). Drop the globe swipe button entirely. The spec already notes this in Section 11 as the simpler approach.

### Recommend cutting: Phone icon in Settings
- Shows a phone icon next to the language matching OS locale
- Low value — users know their device language
- Adds `expo-localization` dependency to the Settings screen
- Cut it for V1

### Keep as-is (not overengineered)
- Tag localization system — O(1) lookup, necessary for bilingual UX
- Search normalization with all language variants — core requirement
- Polish classification keywords — minimal change, leverages existing tokenization
- Separate UI language and OCR language preferences — distinct user needs
- `PreferencesProvider` at top of provider chain — correct dependency order

## Risk Areas

1. **Tesseract.js Polish quality** (Open Question #3) — Polish diacritics may produce poor results. Test early in Phase 4.
2. **Worker reinit latency** — ~2s + ~4MB download on first Polish OCR. Consider if this needs a loading indicator.
3. **Tag input localization direction** — When users type tags in the edit form, they type canonical English keys (`#pasta`). The spec doesn't require Polish tag input resolution (correctly listed as Open Question #2 for future enhancement).
